## Overview

Rebrand the project to **WebPulse AI** ‚Äî a multi-tool web diagnostics platform with a tools hub home page and five independent features accessible via dedicated routes.

**Tagline:** *Pulse-check the web ‚Äî sites, speed, DNS, SSL & uptime.*

## Goals

- Rebrand to WebPulse AI across UI, metadata, README, and PDF footer
- Tools hub at `/` with card navigation to each feature
- Add four new tools: My Internet Speed (full test), DNS Lookup, SSL Check, Uptime/Ping
- Keep existing full website analysis (move to `/analyze`)
- Modular NestJS backend + Next.js routes per tool

## Non-Goals (v1)

- Website instant speed test (HTTP/TTFB for URLs) ‚Äî not selected for v1
- User accounts, saved history, or dashboards over time
- Third-party paid APIs for DNS/SSL
- Mobile native apps
- AI/LLM integration beyond existing rule-based summary on `/analyze`

### 2. My Internet Speed (`/internet-speed`)

**Architecture:** Browser-orchestrated test; backend provides measurement endpoints.

#### Backend endpoints

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/speed-test/ping` | Returns `{ timestamp }` for latency measurement |
| GET | `/speed-test/download?size=1mb\|5mb` | Returns binary payload (Cache-Control: no-store) |
| POST | `/speed-test/upload` | Accepts raw body; returns `{ receivedBytes }` |

Payload sizes:
- Download: 1 MB and 5 MB buffers generated in memory
- Upload: client sends 1 MB and 5 MB blobs

Rate limiting (basic): max 10 speed-test requests per IP per minute (in-memory map, v1).

#### Frontend flow

1. User clicks **Start Test**
2. **Ping phase:** 10 sequential `GET /speed-test/ping` ‚Üí compute avg latency (ms) and jitter (standard deviation of samples)
3. **Download phase:** fetch 1 MB then 5 MB ‚Üí compute Mbps from `(bytes * 8) / timeMs`
4. **Upload phase:** POST 1 MB then 5 MB blobs ‚Üí compute Mbps
5. Display results: Download, Upload, Ping, Jitter

#### Result shape

```typescript
interface InternetSpeedResult {
  downloadMbps: number;
  uploadMbps: number;
  pingMs: number;
  jitterMs: number;
  testedAt: string;
}
```

All computation on frontend; backend only serves payloads.

### 3. DNS Lookup (`/dns`)

**Endpoint:** `POST /dns/lookup`

Request: `{ "domain": "example.com" }`

Uses Node.js `dns.promises`:
- `resolve4` (A)
- `resolve6` (AAAA)
- `resolveMx` (MX)
- `resolveCname` (CNAME)
- `resolveNs` (NS)
- `resolveTxt` (TXT)
- `resolveSoa` (SOA)

Response:
```typescript
interface DnsLookupResult {
  domain: string;
  records: { type: string; values: string[] }[];
  queriedAt: string;
}
```

Validation: strip protocol/path from input; domain format check. SSRF N/A (DNS only, no HTTP to user IP).

Errors: `ENOTFOUND`, `ENODATA` ‚Üí partial empty records + message, not 500.

### 4. SSL Certificate Check (`/ssl`)

**Endpoint:** `POST /ssl/check`

Request: `{ "domain": "example.com" }`

Uses `tls.connect({ host, port: 443, servername: domain })`:
- `valid`: cert dates vs now
- `issuer`, `subject`
- `validFrom`, `validTo`
- `daysRemaining`
- `protocol` (TLS version)
- `altNames` (SAN count or list, capped at 10 for display)

Timeout: 10 seconds.

### 5. Uptime / Ping (`/uptime`)

**Endpoint:** `POST /uptime/check`

Request: `{ "url": "https://example.com" }`

Reuses `HttpFetcherService` (URL validation + SSRF guard):
- `up`: statusCode < 500
- `statusCode`
- `responseTimeMs` (from fetch timing.totalTimeMs)
- `finalUrl`

Optional: use HEAD request variant if added to fetcher; otherwise GET with timing is sufficient.

## Security

- SSRF guard on `/analyze` and `/uptime` (existing)
- DNS/SSL: validate domain string only; no private IP resolution bypass for HTTP
- Speed test: rate limit per IP; cap upload body at 6 MB
- CORS unchanged (frontend origin only)
