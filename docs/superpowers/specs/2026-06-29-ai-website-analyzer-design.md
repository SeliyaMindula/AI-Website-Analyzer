# AI Website Analyzer — Design Spec

**Date:** 2026-06-29  
**Status:** Approved  
**Goal:** Portfolio / learning project

## Overview

A full-stack web tool where a user enters a URL and receives a unified analysis report covering SEO, speed, security headers, tech stack detection, and an AI-style summary with actionable recommendations. Users can download a formatted PDF report generated by the NestJS backend.

## Goals

- Demonstrate full-stack TypeScript skills (Next.js + NestJS)
- Ship locally first with production-ready env configuration
- Run all analysis server-side with no paid API dependencies by default
- Keep architecture modular so analyzers and summary providers are independently testable
- Generate downloadable PDF reports server-side

## Non-Goals (v1)

- User accounts or saved report history
- Progressive/streaming results (SSE)
- Self-hosted Lighthouse fallback
- Paid LLM integration (interface only; rule-based default)

## Architecture

### Stack

| Layer | Technology |
|-------|------------|
| Frontend | Next.js (App Router), TypeScript, Tailwind CSS |
| Backend | NestJS, TypeScript, PDFKit |
| Repo layout | Minimal monorepo: `frontend/` + `backend/` |
| Database | None (stateless) |

### Data Flow

1. User enters URL on Next.js frontend and clicks **Analyze**
2. Frontend sends `POST /analyze { url }` to NestJS
3. NestJS validates URL and blocks SSRF targets (localhost, private IPs)
4. NestJS runs all analyzers in parallel via `Promise.allSettled`
5. Summary provider generates overview and recommendations from combined results
6. NestJS returns unified `AnalysisReport` JSON
7. Frontend renders results dashboard
8. User clicks **Download PDF** → frontend sends report JSON to `POST /report/pdf` → NestJS returns PDF file → browser saves it

```
User → Next.js → NestJS → [Target Site + PSI API]
                ↓
         AnalysisReport JSON → Next.js dashboard
                ↓
         POST /report/pdf → PDF file download
```

## Backend Design

### API

**`POST /analyze`**

Request:
```json
{ "url": "https://example.com" }
```

Response: `AnalysisReport` (see Types below)

**`POST /report/pdf`**

Request: full `AnalysisReport` JSON (same shape returned by `/analyze`)

Response:
- `Content-Type: application/pdf`
- `Content-Disposition: attachment; filename="website-analysis-{hostname}-{date}.pdf"`
- Body: PDF binary stream

Validation:
- Reject malformed or incomplete report bodies with `400`
- Max request body size: 500 KB (prevents abuse; typical report is < 50 KB)

### PDF Report Generation

`PdfGeneratorService` uses **PDFKit** to build a structured, print-friendly document from `AnalysisReport`:

1. **Cover section** — project title, analyzed URL, date, overall grade badge
2. **Summary** — overview paragraph + numbered recommendations (priority labeled)
3. **SEO** — score, key findings, issue list
4. **Speed** — performance score, Core Web Vitals table, top opportunities
5. **Security** — header checklist with present/missing status
6. **Tech stack** — detected technologies grouped by category

Formatting rules:
- A4 page size, consistent margins
- Section headings with horizontal rules
- Page breaks before each major section if content overflows
- Graceful handling of section errors (e.g. "Speed analysis unavailable: …")
- Footer on each page: "Generated by AI Website Analyzer" + page number

`ReportService` pipes PDFKit output to the HTTP response stream (no temp files on disk).

### HttpFetcherService

Shared HTTP layer used by SEO, security, and tech analyzers:

- Normalize URL (prepend `https://` if no scheme)
- SSRF protection: reject `localhost`, `127.0.0.1`, `0.0.0.0`, and RFC 1918 private ranges
- Request timeout: 10 seconds
- Max redirects: 5
- Return: `{ html, headers, finalUrl, statusCode }`

### Analyzer Modules

#### SeoAnalyzer

Parses HTML with Cheerio. Returns:

- `title` (present, length, content)
- `metaDescription` (present, length, content)
- `canonical` URL
- `robots` meta / header
- `h1` count and text list
- Open Graph tags (`og:title`, `og:description`, `og:image`)
- `html[lang]` attribute
- Image `alt` coverage (images with/without alt)
- `score` (0–100) and `issues[]` with severity (`error` | `warning` | `info`)

#### SecurityAnalyzer

Reads response headers. Checks presence and quality of:

- `Content-Security-Policy`
- `Strict-Transport-Security`
- `X-Frame-Options`
- `X-Content-Type-Options`
- `Referrer-Policy`
- `Permissions-Policy`

Returns `score`, per-header status (`present` | `missing` | `weak`), and `issues[]`.

#### TechStackAnalyzer

Detects technologies from headers, HTML patterns, and script/link tags:

- CMS (WordPress, Shopify, etc.)
- Frameworks (React, Next.js, Vue, Angular)
- CDN (Cloudflare, etc.)
- Analytics (Google Analytics, GTM)

Uses lightweight heuristics (no paid Wappalyzer API). Returns `technologies[]` with `name`, `category`, and `confidence` (`high` | `medium` | `low`).

#### SpeedAnalyzer

Calls Google PageSpeed Insights API (mobile strategy). Returns:

- `performanceScore` (0–100)
- Core Web Vitals: `lcp`, `cls`, `inp` (with pass/fail thresholds)
- Top 3 opportunities (title + estimated savings)
- `error` field if PSI call fails

### Summary Provider

```typescript
interface SummaryProvider {
  generate(report: RawAnalysisReport): Promise<AnalysisSummary>;
}
```

**RuleBasedSummaryProvider (default):**

- Computes overall grade (`A`–`F`) from weighted section scores
- Generates 1–2 sentence overview
- Produces top 3–5 prioritized recommendations from highest-severity issues across all sections
- No external API calls

**Future providers:** `OpenAISummaryProvider`, `AnthropicSummaryProvider` — selected via `SUMMARY_PROVIDER` env. Stubs included in v1; implementation deferred.

### Orchestration

`AnalyzeService` runs analyzers in parallel:

```typescript
const [seo, security, tech, speed] = await Promise.allSettled([
  this.seoAnalyzer.analyze(fetchResult),
  this.securityAnalyzer.analyze(fetchResult),
  this.techStackAnalyzer.analyze(fetchResult),
  this.speedAnalyzer.analyze(url),
]);
```

- Fetch HTML/headers once, share across SEO/security/tech
- Speed analyzer calls PSI independently (uses URL, not HTML)
- Failed analyzers return `{ error: string }` in their section; other sections still render
- Global request timeout: 60 seconds

## Frontend Design

### Routes

| Route | Description |
|-------|-------------|
| `/` | URL input form + results dashboard (single-page flow) |

### UI Layout

1. **Header** — project name and one-line description
2. **URL input** — text field + Analyze button
3. **Loading state** — spinner with message: "Analyzing… speed checks can take up to 30 seconds"
4. **Results dashboard** (shown after response):
   - Overall grade badge + summary card (overview + recommendations)
   - **Download PDF** button (top-right of results; disabled until report loads)
   - 4-column grid (2×2 on tablet, stacked on mobile):
     - SEO card (score, issue list)
     - Speed card (score, CWV metrics, opportunities)
     - Security card (header checklist with pass/fail icons)
     - Tech stack card (technology badges by category)

### PDF Download Flow

1. User clicks **Download PDF** after results are displayed
2. Frontend shows brief loading state on the button ("Generating…")
3. Frontend `POST`s the in-memory `AnalysisReport` to `/report/pdf`
4. Frontend receives blob response and triggers browser download via temporary `<a>` element
5. On failure, show toast/error: "Could not generate PDF. Please try again."

### Visual Style

- Dark theme by default
- Score color coding: green (≥80), amber (50–79), red (<50)
- Expandable issue lists per section
- Section-level error cards when an analyzer fails

### Frontend State

- `useState` for `url`, `loading`, `report`, `error`, `pdfLoading`
- No global state library required

## Types

```typescript
interface AnalysisReport {
  url: string;
  analyzedAt: string; // ISO 8601
  summary: AnalysisSummary;
  seo: SeoResult | SectionError;
  security: SecurityResult | SectionError;
  speed: SpeedResult | SectionError;
  techStack: TechStackResult | SectionError;
}

interface AnalysisSummary {
  grade: 'A' | 'B' | 'C' | 'D' | 'F';
  overview: string;
  recommendations: Recommendation[];
}

interface Recommendation {
  priority: 'high' | 'medium' | 'low';
  category: 'seo' | 'security' | 'speed' | 'tech';
  message: string;
}

interface SectionError {
  error: string;
}
```

## Error Handling

| Scenario | HTTP Status | User-facing message |
|----------|-------------|---------------------|
| Malformed URL | 400 | "Please enter a valid URL" |
| SSRF / private IP | 400 | "Cannot analyze local or private addresses" |
| Target unreachable | 200 (partial) | SEO/security/tech sections show section error |
| PSI API failure | 200 (partial) | Speed section shows section error |
| Global timeout (60s) | 504 | "Analysis timed out. Try again." |
| Unhandled server error | 500 | "Something went wrong. Please try again." |
| Invalid report body for PDF | 400 | "Invalid report data" |
| PDF generation failure | 500 | "Could not generate PDF. Please try again." |
